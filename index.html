Why this runtime exists

We are building a multi-agent graph runtime around a simple architectural distinction. An environment is not a loose bag of prompts, tools, and context windows. It has structure. It consists of graph structure plus small neural networks. An agent is an environment with an LLM.

Environment = graph structure + small neural networks
Agent = environment + LLM

This matters because many current AI builders multiply agents before defining agency. A system is called multi-agent simply because several large models or chat endpoints are exchanging messages. But real coordination requires more than traffic. It requires representation.

In a serious multi-agent system, each agent should track not only the task, but what other agents know and what they think fellow agents know about the task. This improves coordination first. It also opens the way to stronger verification later, because explicit structure can be inspected, traced, and checked.

Coordination improves when agents model each other, not just message each other.

Graph structure is central here. A graph makes it possible to represent environments, relations, task states, and agent models in explicit form rather than leaving everything dissolved inside undifferentiated text.

Layered intelligence, not one giant black box

Our approach is not based on a single model doing everything by itself. Some parts of the system require high-level orchestration. Some require narrow learned behavior. Some require explicit graph structure. Some require representations of other agents and of the environment they share.

This is why the runtime is not merely an LLM wrapper and not merely an orchestration layer. It is an attempt to build a more serious architecture for agent-based intelligence. The smartest model should orchestrate. Smaller specialized neural systems should do narrower work.

Reinforcement learning is one of the pillars, but not only in the familiar “LLM plus RL” sense. We also treat RL-trained Graph Neural Networks as foundational, because structured environments must be learned and optimized as structured environments, not merely described in language.

Current AI builders often multiply agents before defining agency.

Toward teachable and verifiable competence

We do not treat Agentic Form as a totally unknown latent that must emerge blindly from opaque training. Important aspects of agentive internal structure are already partly knowable through action, coordination, and advanced semantics.

The result we are aiming for is not just smarter output. It is a system in which competence can be carried, reused, and eventually taught. An agent that carries an environment does not merely execute a capability. It carries structured competence with it.

You only know what you can teach.

Over time, that same explicit structure may support stronger automatic verification, including workflows where legal and commercial trust matter.

Why now

This product is being built now because earlier generations of models were too weak to help construct it fast enough. They could describe fragments of the vision, but not participate effectively in building the runtime itself. That has changed.

Current models are finally strong enough to help implement the architecture that goes beyond ordinary prompt-centric workflows. In that sense, today’s strongest models are not the final architecture. They are participants in building the next one.
